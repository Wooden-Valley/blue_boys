# Методичка: совместная работа в Git

## 1. Настройка (делается один раз на компьютере)

```bash
# Указываем имя — оно будет в подписи каждого коммита
git config --global user.name "Иван Иванов"

# Указываем почту (ту же, что на GitHub)
git config --global user.email "ivan@example.com"

# Проверить, что записалось
git config --list
```

---

## 2. Начало работы с проектом

### Вариант А. Проект уже есть на GitHub — копируем его себе

```bash
# Скачать (склонировать) репозиторий на свой компьютер
git clone https://github.com/имя-команды/название-проекта.git

# Зайти в скачанную папку
cd название-проекта
```

### Вариант Б. Создаём репозиторий из своей папки

```bash
# Превратить текущую папку в git-репозиторий
git init

# Привязать к удалённому репозиторию на GitHub (origin — его стандартное имя)
git remote add origin https://github.com/имя-команды/название-проекта.git
```

---

## 3. Главные команды на каждый день

```bash
# Посмотреть, что изменилось и в каком состоянии файлы
git status

# Забрать свежие изменения товарищей с GitHub (ДЕЛАЙ ЭТО ПЕРЕД НАЧАЛОМ РАБОТЫ!)
git pull

# Добавить конкретный файл к будущему коммиту
git add имя_файла.txt

# Добавить ВСЕ изменённые файлы
git add .

# Сохранить изменения в историю с понятным описанием
git commit -m "Добавил страницу регистрации"

# Отправить свои коммиты на GitHub, чтобы их увидела команда
git push
```

> Запомни порядок на каждый раз: **pull → add → commit → push**

---

## 4. Ветки (branches) — чтобы не мешать друг другу

Ветка — это отдельная линия работы. Каждый делает свою задачу в своей ветке, потом всё объединяется.

```bash
# Посмотреть список веток (звёздочка — текущая)
git branch

# Создать новую ветку и сразу перейти в неё
git checkout -b моя-задача

# Переключиться на существующую ветку
git checkout main

# Отправить свою новую ветку на GitHub
git push -u origin моя-задача
```

> `main` (или `master`) — главная ветка проекта. Старайтесь не ломать её: работайте в своих ветках, а объединение делайте через Pull Request на сайте GitHub.

---

## 5. Объединение работы (merge)

```bash
# Перейти в главную ветку
git checkout main

# Забрать свежую версию
git pull

# Влить в неё свою ветку
git merge моя-задача
```

На практике чаще объединяют через **Pull Request** на GitHub: создаёте PR из своей ветки в `main`, товарищи смотрят код и нажимают «Merge».

---

## 6. Конфликты (conflict) — не пугайтесь

Конфликт возникает, когда двое поменяли одну и ту же строку. Git не знает, чей вариант оставить, и помечает место так:

```
<<<<<<< HEAD
твой вариант строки
=======
вариант товарища
>>>>>>> другая-ветка
```

Что делать:

1. Открыть файл, найти эти метки.
2. Оставить правильный текст, **удалить** строки `<<<<<<<`, `=======`, `>>>>>>>`.
3. Сохранить файл и закоммитить:

```bash
git add имя_файла
git commit -m "Решил конфликт"
```

> Лучшая защита от конфликтов — почаще делать `git pull` и не работать всем в одном файле.

---

## 7. Полезное

```bash
# История коммитов (q — выйти)
git log --oneline

# Отменить изменения в файле (вернуть как было до правок)
git checkout -- имя_файла

# Посм, какие именно строки изменились
git diff
```

---

## 8. Правила хорошего тона в команде

- 🔄 Перед работой всегда делай `git pull`.
- 💬 Пиши понятные сообщения коммитов: «Исправил ошибку входа», а не «фикс».
- 🧩 Делай маленькие коммиты — их проще проверять и откатывать.
- 🌿 Каждую задачу — в отдельной ветке.
- 🙋 Не уверен — спроси команду перед `git push` в `main`.
